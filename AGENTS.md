# AGENTS.md

Orientation for AI coding agents working in this repository. Pair this with
[`ARCHITECTURE.md`](./ARCHITECTURE.md) — that file is the source of truth for
_how_ the codebase is built. This file covers _how to work in it_ without
duplicating that content.

If anything here conflicts with `ARCHITECTURE.md`, the architecture doc wins.

---

## What this project is

A Flutter template with a strict 3-layer architecture (`data` / `domain` / `ui`),
designed to be copied into new apps. The shape is opinionated and small.
Read [`ARCHITECTURE.md`](./ARCHITECTURE.md) before making non-trivial changes.

## Setup

```bash
make setup       # fvm install (pulls Flutter 3.44.1), pub get, lefthook install
make run         # uses persisted env or `prod` as fallback
```

> **Important — two pieces of tooling friction:**
>
> 1. **Flutter version is pinned to 3.44.4 via FVM** (`.fvmrc`). All Flutter
>    and Dart commands must be routed through `fvm` (`fvm flutter`,
>    `fvm dart`). The Makefile and lefthook do this automatically.
> 2. **Primary constructors are an experimental Dart feature** requiring
>    `--enable-experiment=primary-constructors` on every `flutter` / `dart run`
>    invocation. The Makefile passes it. `fvm dart analyze` is the exception:
>    it reads the flag from `analysis_options.yaml`.
>
> Prefer the `Makefile` targets (`make run`, `make test`, `make analyze`,
> `make layers`, `make check`) over raw `flutter`/`dart` invocations.

## Build, analyze, test

```bash
make analyze       # fvm dart analyze --fatal-infos
make format        # fvm dart format ... .
make test          # fvm flutter test --enable-experiment=primary-constructors
make layers        # fvm dart run tool/check_architecture_violations.dart
make check         # all of the above in one go
```

Pre-commit and pre-push hooks run analyze + format + the layer-boundary check;
pre-push additionally runs `flutter test`. Skip with `git push --no-verify` for
WIP pushes.

## Code style — essentials

Full conventions live in [`ARCHITECTURE.md`](./ARCHITECTURE.md). The
high-frequency rules an agent will hit:

- **No `dynamic`. No `double`.** Strong types everywhere; use inference where it
  doesn't obscure intent.
- **Primary constructors** for DTOs, entities, value classes, and any class
  whose fields all map 1:1 to constructor parameters. Example:
  `class User(final String id, final String name) extends Equatable { ... }`.
- **Immutable by default.** All entities, DTOs, and notifier state are
  immutable. `Equatable` for equality; manual `copyWith`.
- **No codegen** (one exception: Flutter's built-in `gen-l10n`). No Freezed,
  no json_serializable, no go_router_builder, no mockito. Manual JSON, manual
  copyWith, mocktail for tests.
- **Localization:** `L10n.translate.<key>` in widgets, ARB files in
  `assets/l10n/`, generated `AppLocalizations` in
  `lib/ui/shared/l10n/generated/` (gitignored; regenerated by
  `fvm flutter pub get`). Never call `AppLocalizations.of(context)` directly —
  go through `L10n`.
- **Single DI helper module:** `lib/shared/inject.dart` exports `inject<T>()`
  (typed lookup) and `injector` (raw GetIt instance, for registration). Never
  use `GetIt.I` directly — that's only allowed inside `inject.dart` itself.
- **Logging:** `log.info(…)` / `log.error(…)` via the global in
  `lib/shared/log.dart`. Do not call `inject<ILogger>()` directly.
- **Single quotes**, trailing commas in multi-line arg lists, prefer `const`.

## Layer rules (enforced)

```
ui     → may import domain, shared
domain → may import data, shared
data   → may import shared only
shared → may import nothing inside lib/
```

Violations are caught by `tool/check_architecture_violations.dart` and block commits.
Additional rules enforced by the same script:

- `lib/ui/features/**` may not import `package:get_it/**` or use the literal
  text `inject<` — go through Riverpod providers/notifiers, or the `log` global.

## Where to put new code

- **A new screen** → `lib/ui/features/<feature>/<feature>_page.dart`. Define
  `path`, `name`, `static GoRoute route()`, and (if parameterized) a
  `static void go(...)` helper on the page. Add the page's `.route()` to
  `lib/ui/shared/router/app_routes.dart`.
- **A new use-case** → `lib/domain/use_cases/<verb_noun>.dart`. Implement
  `UseCase<R>` or `StreamUseCase<R>`. Per-call params go in the constructor;
  internal dependencies via `inject<T>()` getters. Do **not** register the
  use-case in GetIt.
- **A new repository** → `lib/data/repositories/<x>_repository.dart`. Concrete
  class. Register in `data_module.dart`. Return DTOs.
- **A new DTO** → `lib/data/models/<x>_dto.dart`. Primary constructor,
  `factory fromJson`, manual `toJson()` when writes are needed.
- **A new entity** → `lib/domain/entities/<x>.dart`. Primary constructor,
  `extends Equatable`, manual `copyWith`. Companion converter in
  `lib/domain/converters/<x>_converter.dart` as an extension on the DTO.
- **State for a single feature** → inside that feature's folder
  (`lib/ui/features/<x>/<x>_notifier.dart`). Promote to
  `lib/ui/shared/state/` only when a second feature consumes it.
- **A platform-service wrapper** → `lib/data/services/<x>_service.dart`. Define
  an `IXService` interface when an in-memory test impl is plausible; otherwise
  keep concrete-only.

## Riverpod ↔ hooks

- Widgets extend `HookConsumerWidget`.
- **Shared state** → Riverpod (`FutureProvider`, `AsyncNotifierProvider`,
  always `autoDispose` unless documented otherwise).
- **Local widget state** → `flutter_hooks` (`useTextEditingController`,
  `useState`, …).
- Providers call use-cases; widgets do not call use-cases directly.

## Errors

- The dio interceptor translates HTTP errors into the sealed `DataException`
  hierarchy. Repos do not write try/catch.
- Use-cases let `DataException` propagate by default; catch + rethrow as a
  `DomainException` only when domain semantics differ.
- UI surfaces errors via `AsyncValue.when(error: …)` with `switch` over the
  sealed types. `errorMessage(e)` in `lib/ui/shared/error_messages.dart` is
  the central exception → user-string mapper.

## Testing

- `mocktail` for mocks.
- Mirror `lib/` in `test/`.
- Tests that hit `inject<T>()` start with `setUp(setupTestDi)` (helper in
  `test/helpers/di_test_helper.dart`) which resets GetIt and registers
  in-memory / mock impls. `InMemoryPreferenceService` is the default
  `IPreferenceService` in tests.
- UI tests override Riverpod providers via `ProviderScope(overrides: [...])`.
- Shared fixtures in `test/helpers/fixtures.dart`.

## Commits & PRs

- Conventional-ish messages: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`,
  `chore:`. Imperative mood. Keep the subject ≤ 72 chars.
- Each commit should leave `flutter analyze`, `flutter test`, and the
  layer-boundary check passing. Hooks enforce this on push.
- PR descriptions should call out any deviation from `ARCHITECTURE.md` and
  justify it. If you are tempted to deviate, prefer a doc-update PR first so
  the deviation lands in the architecture rather than as drift.

## Things to avoid

- Adding codegen-based packages (Freezed, json_serializable, retrofit, etc.).
  If you reach for one, stop and reconsider the manual approach.
- Calling `inject<T>()` inside widgets in `lib/ui/features/`.
- Using `GetIt.I` directly anywhere except inside `lib/shared/inject.dart`.
  For registration use `injector.registerSingleton<T>(...)`; for lookup use
  `inject<T>()`.
- Introducing abstract repository interfaces "just for testability" — GetIt
  registration is enough. Interfaces are reserved for services with a real
  second implementation (`IPreferenceService`, `ILogger`).
- Centralizing route constants — pages own their routes.
- Adding `late` fields on notifier state. State is immutable.
- Using `dynamic` or `double` to make a type problem go away.

## Adding a feature: short checklist

1. New feature folder under `lib/ui/features/<feature>/`.
2. DTO(s) under `lib/data/models/`; repo method(s) on a new or existing repo.
3. Entity in `lib/domain/entities/`; converter extension in
   `lib/domain/converters/`.
4. Use-case class in `lib/domain/use_cases/`.
5. Riverpod provider/notifier inside the feature folder.
6. Page(s) with `route()` + add to `app_routes.dart`.
7. Tests at each layer (`test/data/`, `test/domain/`, `test/ui/`).
8. Run `dart run tool/check_architecture_violations.dart`, `flutter analyze`,
   `flutter test`.

## When unsure

Read `ARCHITECTURE.md`. If the answer isn't there, ask before improvising —
template drift is the cheapest thing to prevent and the most expensive thing
to undo.
